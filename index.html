Hardware Triage & Health-Check Workflow (Windows)

Objective: I inspected hardware health, verified BIOS and diagnostics, and documented findings as a data center technician would during a hardware triage incident.

Steps Followed

1. Record system details
  • Device model and serial/service tag
  • CPU and memory configuration
  • Disk type and capacity
  • Installed OS and firmware version
2. Enter BIOS / UEFI
  • Restart and press F2 (Dell) or F12 for diagnostics menu
  • Verify boot mode, secure boot state, and detected disks
  • Record whether the internal drive and RAM are detected
3. Run manufacturer diagnostics
  • Boot Dell diagnostics via F12
  • Check memory, disk, fan, and temperature status
  • Note any warnings, failures, or error codes
4. Review system logs
  • Windows: Event Viewer → Windows Logs → System
  • Look for disk/controller errors, unexpected shutdowns, thermal warnings
  • Identify repeated critical events, not one-off warnings
5. Confirm device manager status
  • Open Device Manager
  • Check for missing drivers, yellow warnings, unknown devices
  • Determine whether the issue is hardware, firmware, or driver-related

Outcome

  • Validated whether the machine was operational or required repair
  • Captured a structured triage log of hardware detection and diagnostics
  • Prepared a concise incident summary for resume-ready documentation

Hardware Triage & Health-Check Workflow (Linux)

Objective: I used Linux command-line tools to perform hardware inventory, validate component detection, check disk and sensor health, and documented findings.

Steps Followed

1. Record hardware inventory

Run the following commands to gather system information:

Linux Server Hardware Validation Workflow

1. Confirm kernel and OS state
To get kernel information: uname -a
To get OS release information: cat /etc/os-release

2. Confirm system identity
To get manufacturer, model, and serial number: sudo dmidecode -t system

3. Verify processor detection
To get processor details: lscpu

4. Verify memory detection
To get memory information: free -h

5. Verify storage detection
To list block devices: lsblk
To verify disk detection and partition visibility: sudo fdisk -l

6. Verify disk health
To check disk health: sudo smartctl -a /dev/{disk-name}
Requires the smartmontools package

7. Verify PCI hardware detection
To check PCI hardware detection: sudo lspci

8. Verify loaded kernel drivers
To check loaded drivers/modules: lsmod

9. Check thermal and fan sensors
To detect available sensors: sudo sensors-detect
To display temperature and fan information: sensors
Requires the lm-sensors package

10. Review hardware and kernel events
To view kernel messages: dmesg | less
To view system logs and events: sudo journalctl

Outcome

  • Performed structured Linux server hardware validation including system identity, CPU, memory, storage, PCI device detection, and kernel driver verification
  • Executed disk health diagnostics using SMART data and validated block-device enumeration at kernel level
  • Reviewed thermal sensors, hardware telemetry, and boot-time kernel events to confirm operational stability
  • Conducted log-based fault triage using dmesg and journalctl and documented results in data center–style incident closure format

Host Network Configuration & Addressing Validation

Objective: I verified how DHCP and static IPv4 addresses behave, and troubleshot connectivity by changing the host addressing mode.

Steps Followed

1. Check current addressing
ipconfig /all

Look for:

  • DHCP Enabled: Yes
  • Assigned IPv4 address
  • Gateway and DNS settings
2. Switch to static mode

Ethernet → Properties → IPv4

Set:

  • IP: 192.168.1.50
  • Subnet: 255.255.255.0
  • Gateway: 192.168.1.1
  • DNS: 8.8.8.8
3. Test connectivity
ping 192.168.1.1
ping google.com
4. Return to DHCP
ipconfig /renew

Confirm a new DHCP address is assigned

Outcome

  • Confirmed IP assignment behavior for DHCP and static addressing
  • Proved local gateway and external connectivity checks
  • Captured evidence of configuration-related connectivity issues

Layered Network Path Isolation (Host → Switch → Router → Internet)

Objective: I systematically isolated network faults from physical link issues through gateway, external IP, and DNS testing.

Steps Followed

1. Check physical failure

Ethernet

ipconfig

Expected: 192.168.254.*

If "Media disconnected", then check: Cable, Port, NIC status

2. Check IP address
ipconfig /all

Expected: Details as in the below picture, it means DHCP is working, Device is connected to network

If "169.254.x.x", then DHCP failure, no network communication

3. Subnet correctness
Check if the IP address is within the correct subnet range

Expected: 255.255.255.0, means host can reach gateway

If mismatch, the host cannot reach the gateway and local routing will fail

4. Test TCP/IP stack routing
ping 127.0.0.1

Expected: Successful ping to localhost means TCP/IP stack is functioning

If ping fails, it indicates a local host configuration issue or TCP/IP stack problem

5. Gateway reachability (local network validation)
ping 192.168.1.1

Expected: Successful ping to gateway means local network connectivity is working

Also means Cable, Switch, VLAN, Local Routing is functioning

If ping fails, it indicates a local network issue such as switch port failure, VLAN misconfiguration

6. External routing (internet reachability)
ping 9.9.9.9

Expected: Successful ping to external IP means internet connectivity is working

Also means Cable, Switch, VLAN, Local Routing is functioning

If ping fails, it indicates an issue with upstream routing or DNS configuration

Also if gateway is misconfigured

7.DNS resolution
nslookup google.com

Expected: Successful DNS resolution means domain names can be translated to IP addresses

Also means Cable, Switch, VLAN, Local Routing is functioning

If resolution fails, it indicates a DNS configuration issue

Fix the DNS settings

Outcome

  • Confirmed whether the fault was physical, local network, upstream routing, or DNS
  • Applied layered troubleshooting exactly like a DC technician
  • Created a clear resolution path for common network outages
